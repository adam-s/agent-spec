### Key Level 0 lessons

1. **Observe before concluding.** Two "failures" were broken verify.sh scripts. I declared "instructions aren't any good" without looking at traces. The human had to push me to observe.
2. **Synthetic prompts lose signal.** Real bug reports carry the natural level of specificity reporters provide. Rewriting adds bias in both directions.
3. **Instructions help but have limits.** Some failures are model capability thresholds, not instruction gaps. Escalation to stronger models is the right response.
4. **The reviewer automates diagnosis.** The conversation about "what did the agent miss" is exactly what the reviewer does with structured output.

## What's next

### 1. Run the full iterate loop

Use `/iterate` on bug-squashing with the reviewer agent. The loop should: run all challenges → reviewer diagnoses failures → apply generalized findings → re-run. This is what we did manually this session, automated.

### 2. Budget ladder

Run the same bugs at different token budgets to map tokens-to-correctness curves. As instructions improve, the curve should shift left (fewer tokens for same result).

### 3. Harder bugs

All 7 haiku-passable challenges pass on first attempt. Need bugs where the instructions make the difference — bugs that fail without good instructions and pass with them. The marshmallow case showed this (FAIL→PASS from one instruction), but we need more.

### 4. Multi-agent debugging

The escalation ladder currently goes haiku → sonnet. A middle step: haiku fails → reviewer analyzes the failure → feed the finding back as additional context → retry haiku with the hint. This is cheaper than sonnet and tests whether the agent just needed a nudge.

### 5. Botocore as the hard benchmark

This bug (preserve-then-delete pattern) is genuinely hard. Use it as the test case for multi-agent debugging and reviewer feedback loops. If we can make haiku solve it with reviewer assistance, the instructions and reviewer are working together.
