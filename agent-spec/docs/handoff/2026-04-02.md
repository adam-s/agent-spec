# Handoff — 2026-04-02

## What agent-spec is

The product is `agent-spec/.claude/` — orchestrator instructions that make an agent capable of setting up and running ANY experiment a developer describes. Scripts are utilities. Evals are ephemeral test output. The `.claude/` is the product.

## What we did today

### Major architectural changes

1. **Rewrote CLAUDE.md** — three-layer testing model (output, config, behavior), recursive architecture (Level 0/1/2), deterministic termination principle, self-improvement loop, product definition
2. **Built component design framework** — 8 reference docs distilled from official Claude Code documentation (hooks, rules, permissions, settings, skills, agents, decision tree, component design). These are diagnostic tools for scoring `.claude/` directories. Live in `.claude/reference/`.
3. **Applied decision tree reflexively** — moved 248 lines from always-loaded rules to on-demand reference docs. Added `permissions.deny` to all target configs. Eliminated all "NEVER overwrite" prose.
4. **Renamed targets/ → evals/, target.yaml → EVAL.md** — unified format following SKILL.md/AGENT.md pattern. YAML frontmatter + markdown body.
5. **Added workspace model** — workspaces can be built from source repos, seed files, or both. Replaced "sandbox" language that locked thinking into "copy a repo."
6. **Added challenges × configs = runs** — one eval can have multiple challenges and multiple configs. The cross-product is the set of runs.
7. **Added observability rule** — every run must produce visible progress, never pipe through filters that lose data.
8. **Added self-improvement loop** — failed experiments are signal about the `.claude/` instructions. Stop, diagnose, fix instructions, delete artifacts, start fresh.

## What's broken right now

### 1. hono-websocket-counter fails all configs

All 6 configs fail with sonnet. The testing-claude-agent repo got these to pass. Need to investigate whether the workspace assembly is missing something (package.json contents? bun lockfile? setup not running?) or if the verify.sh has a problem.

Run IDs for investigation: a914021a, e61b67d3, a3d8fb00, affd62fd, 93af0fb4, 9da864ba

Check with: `jq 'select(.event=="verification_output")' /tmp/agent-spec/<run_id>/events.jsonl`

### 2. Reporting script was broken

The shell loop that ran the 18-cell matrix used grep to find "RESULT: PASS" in invoke.py's stdout. But invoke.py writes results to events.jsonl, not stdout in a greppable way. All 18 runs showed UNKNOWN/$2.00 even though 12 actually passed. The results had to be extracted from events.jsonl after the fact.

The observability rule was added to prevent this but the script was written before the rule took effect. `run_eval.py` needs to print a clear summary line to stdout after each run.

### 3. Scripts still reference old concepts

`scripts/invoke.py` still uses `_sandbox` as an internal variable name (renamed to `workspace_path` for args but internal references mixed). `scripts/parallel.py` doesn't understand the challenges × configs matrix — it only iterates configs for a single eval.

### 4. `list_evals()` won't find config-comparison

The `list_evals()` function in `lib.py` looks for `EVAL.md` files. config-comparison has one, but the function also tries to parse `source:` which doesn't exist in matrix evals. Need to verify `cli.py list` works.
